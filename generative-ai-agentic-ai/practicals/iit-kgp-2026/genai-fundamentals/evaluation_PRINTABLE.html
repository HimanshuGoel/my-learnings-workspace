Topic 12 — Evaluation

Summary: Evaluation measures how well your LLM application works across multiple dimensions — accuracy, faithfulness, relevance, and safety — using automated metrics, LLM-as-judge, and human review.

Why It Exists

  • Problem — unlike traditional ML (clear accuracy/F1), LLM outputs are open-ended, subjective, multi-dimensional. "It seems to work" isn't enough for production.
  • Solution — structured evaluation across dimensions (faithfulness, relevance, correctness, safety) using automated metrics, LLM-as-judge, and human evaluators.
  • Mental Model — Code reviews + automated testing + QA combined. Unit tests = automated metrics. Code review = LLM-as-judge. QA testing = human eval. CI/CD checks = continuous monitoring.
  • Or: Like observability for microservices. You need metrics (Prometheus), logs (structured evaluation), alerts (quality regression detection), and dashboards (evaluation reports).

How It Works

Multiple dimensions, multiple methods, continuous — not a one-time activity.

Evaluation combines automated metrics, LLM-as-judge, and human review across multiple quality dimensions

LLM-as-Judge Pattern

System: You are an expert evaluator. Rate the following response on: 1. Faithfulness (1-5): Is it supported by the context? 2. Relevance (1-5): Does it answer the question? 3. Completeness (1-5): Does it cover all aspects? Context: {retrieved_documents} Question: {user_question} Response: {model_response} Provide scores and brief justification for each.
LLM-as-Judge trade-off: Scalable and consistent, but has biases (prefers verbose answers, may agree with itself). Calibrate against human ratings. Use as screening, not final verdict.
Evaluation is continuous: deploy → monitor → evaluate → improve → redeploy. Not a one-time activity.

Common Misconceptions

WrongCorrect
"Accuracy is enough"LLM outputs are multi-dimensional. Faithfulness, helpfulness, safety ALL matter.
"Automated metrics are sufficient"BLEU/ROUGE miss quality. Need LLM-as-judge or human eval for real quality assessment.
"One test set is enough"Need diverse cases: edge cases, adversarial inputs, multi-lingual, domain-specific.
"Evaluation is one-time"Continuous monitoring is essential. Model behavior changes with updates/data drift.
"If it works in demos, it works in production"Demo queries ≠ real user queries. Always evaluate on real usage patterns.

Top Takeaways

#Insight
1LLM evaluation is multi-dimensional: faithfulness, relevance, correctness, safety, format compliance.
2No single metric is sufficient. Combine automated metrics + LLM-as-judge + human evaluation.
3RAG evaluation has specific metrics: context precision/recall + faithfulness of generated answer.
4LLM-as-Judge is scalable but biased. Calibrate against human ratings. Use as screening, not verdict.
5Evaluation is continuous (like monitoring), not one-time (like a unit test you write and forget).
6"Works in demo" ≠ "works in production." Always evaluate on real, diverse usage patterns.
7Hallucination detection: ground in context (RAG), self-consistency, claim extraction + verification.
