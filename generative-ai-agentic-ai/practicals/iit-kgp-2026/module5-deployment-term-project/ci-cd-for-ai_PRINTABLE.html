Module 5, Topic 8 — CI/CD for AI

Traditional CI/CD asserts deterministic outputs (assert result == expected). AI systems are non-deterministic — the same input may produce different (valid) outputs. You need evaluation-based quality gates instead of exact-match tests: "is the RAGAS faithfulness score above threshold?" not "is the output this exact string?" This topic covers testing pipelines for AI services with GitHub Actions.

What you're deploying: Automated test pipeline that catches regressions in AI quality (not just code bugs) before deploying to production.

AI CI/CD Pipeline

Standard CI/CD + AI quality eval gate. The eval gate is what makes this pipeline AI-aware.

GitHub Actions — AI Pipeline

# .github/workflows/ai-ci.yml name: AI Service CI/CD on: [push, pull_request] jobs: unit-tests: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: {python-version: '3.11'} - run: pip install -r requirements.txt - run: pytest tests/unit/ -v integration-tests: runs-on: ubuntu-latest needs: unit-tests steps: - uses: actions/checkout@v4 - run: pip install -r requirements.txt - run: pytest tests/integration/ -v env: OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }} ai-quality-gate: runs-on: ubuntu-latest needs: integration-tests if: github.ref == 'refs/heads/main' # Only on main (costs money) steps: - uses: actions/checkout@v4 - run: pip install -r requirements.txt - run: python scripts/run_eval.py --threshold 0.85 env: OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }} # Fails if faithfulness < 0.85 or relevancy < 0.80 deploy: runs-on: ubuntu-latest needs: ai-quality-gate if: github.ref == 'refs/heads/main' steps: - run: docker build -t ai-service . - run: docker push registry/ai-service:${{ github.sha }}

Eval Gate Script

# scripts/run_eval.py import json, sys from ragas import evaluate from ragas.metrics import faithfulness, answer_relevancy from datasets import Dataset def run_eval(threshold: float = 0.85): # Load test set (committed to repo) test_data = json.load(open("tests/eval_dataset.json")) # Run pipeline on test set results = evaluate( dataset=Dataset.from_dict(test_data), metrics=[faithfulness, answer_relevancy], ) print(f"Faithfulness: {results['faithfulness']:.3f}") print(f"Relevancy: {results['answer_relevancy']:.3f}") # Quality gate if results['faithfulness'] < threshold: print(f"❌ FAILED: faithfulness {results['faithfulness']:.3f} < {threshold}") sys.exit(1) print("✅ PASSED: AI quality gate") sys.exit(0) if __name__ == "__main__": import argparse parser = argparse.ArgumentParser() parser.add_argument("--threshold", type=float, default=0.85) args = parser.parse_args() run_eval(args.threshold)

Testing Non-Deterministic Outputs

# Strategy: Don't assert exact output. Assert PROPERTIES of the output. def test_rag_response_properties(): """Test that response has expected properties (not exact content).""" response = client.post("/v1/chat", json={"message": "What's the rate limit?"}) body = response.json() # Schema is deterministic assert response.status_code == 200 assert "response" in body assert len(body["response"]) > 20 # Non-trivial answer assert len(body["response"]) < 2000 # Not runaway # Content properties (fuzzy matching) assert any(kw in body["response"].lower() for kw in ["rate", "limit", "request"]) # Source attribution assert body.get("sources") # Should cite sources assert len(body["sources"]) > 0 def test_agent_tool_selection(): """Test that agent uses correct tool (tool choice is more stable than output text).""" # Run 3 times — tool selection should be consistent even if wording varies tools_used = set() for _ in range(3): result = agent.invoke({"messages": [HumanMessage("Calculate 15% of 200")]}) for msg in result["messages"]: if hasattr(msg, "tool_calls") and msg.tool_calls: tools_used.add(msg.tool_calls[0]["name"]) assert "calculate" in tools_used # Should use calculator at least once

Pre-Deploy Checklist

☐ Unit tests for all non-AI code (tools, formatters, validators)
☐ Integration tests: endpoints return correct schema
☐ Eval test set committed to repo (20-50 examples with expected behavior)
☐ Quality gate script: fails CI if faithfulness/relevancy drops
☐ Flaky handling: run eval 3x, pass if 2/3
☐ Smoke test post-deploy (5 known queries return valid response)
☐ Secrets managed (API keys in GitHub Secrets, not in code)
☐ Cost budget for CI eval: estimated and acceptable
☐ Weekly scheduled eval run (catch model drift between deploys)
