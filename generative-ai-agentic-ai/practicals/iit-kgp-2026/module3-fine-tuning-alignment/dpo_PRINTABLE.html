Module 3, Topic 7 — DPO (Direct Preference Optimization)

DPO achieves the same alignment goals as RLHF but reformulates the problem as supervised learning on preference pairs — no reward model, no RL loop, no training instability. Same results, 10× simpler, 50-200× cheaper. This is what you'll actually implement when you need alignment. The key insight: the RLHF reward function can be expressed implicitly through the model's own probabilities, eliminating the need for a separate reward model entirely.

Problem: RLHF is complex (3 models, RL instability, $5K-20K). DPO reformulates alignment as a supervised loss: directly train the model to prefer good responses over bad ones. Same learning outcome, dramatically simpler pipeline.

DPO vs RLHF Architecture Comparison

DPO eliminates the reward model and RL loop entirely. Same alignment quality, dramatically simpler and cheaper.

The DPO Loss — Intuition

# For each training example (prompt, chosen, rejected): # DPO pushes model to: # INCREASE probability of generating the chosen response # DECREASE probability of generating the rejected response # CONSTRAINED by: don't change too much from reference (β parameter) Loss = -log(σ(β × (log_prob_chosen - log_prob_rejected))) # β controls deviation from reference model (same role as KL in RLHF) # With LoRA: base model IS the reference (frozen) → natural constraint!

Implementation — DPO with TRL

Memory-Efficient DPO (LoRA = Reference)

# With LoRA, base model IS the reference (frozen weights) # No need to load a separate ref_model → saves 50% memory! dpo_config = DPOConfig(..., ref_model=None) # Implicit reference via LoRA base # Why this works: LoRA keeps base frozen, only adapter changes # The frozen base naturally provides the KL constraint # This is the RECOMMENDED approach for most practitioners

Preference Data Format & Generation

# Required format: {"prompt": "...", "chosen": "better", "rejected": "worse"} # Generating pairs with GPT-4o as judge: def generate_preference_pair(prompt, sft_model): resp_a = sft_model.generate(prompt, temperature=0.7) resp_b = sft_model.generate(prompt, temperature=0.9) judgment = gpt4o_judge(prompt, resp_a, resp_b) # "A" or "B" if judgment == "A": return {"prompt": prompt, "chosen": resp_a, "rejected": resp_b} return {"prompt": prompt, "chosen": resp_b, "rejected": resp_a} # Existing datasets: UltraFeedback (64K), HH-RLHF (170K), Orca DPO (12K)

β (Beta) Parameter — The Key Knob

β ValueBehaviorWhen to Use
0.05Aggressive — model changes a lotStrong alignment signal, diverse data
0.1Standard default — good balanceStart here for most cases
0.2Conservative — small changesWant subtle improvement only
0.5Very conservative — barely changesWorried about regression
β tuning: If model degrades general capabilities → increase β (more conservative). If alignment is too weak/no visible change → decrease β (allow more change).
