Module 3, Topic 5 — Training Infrastructure

GPU memory is the primary constraint in fine-tuning. You need to match your model size and training method (full SFT vs LoRA vs QLoRA) to available hardware. Cloud GPU options vary wildly in cost — from free (Colab) to $8/hr (H100). This topic helps you pick the right setup without overspending or running out of memory mid-training.

Problem: GPU options are confusing (T4 vs A100 vs H100), cloud providers charge differently, and miscalculating memory requirements means either OOM crashes or paying for unused capacity. You need a clear mental model for hardware selection, cost estimation, and practical cloud GPU workflows.

GPU Memory Budget

Full SFT 8B bf16: 16+16+64+~4 = ~96 GB (needs 2× A100 80GB). LoRA: 16 GB base + tiny adapter overhead = ~18 GB. QLoRA: 4 GB base + adapter = ~8 GB.

Memory Formula Quick Reference

Full SFT (bf16): 12 bytes × param_count → 8B model ≈ 96 GB (2× A100 80GB) LoRA (bf16 base): ~2.5 bytes × param_count → 8B model ≈ 18 GB (1× A100 40GB) QLoRA (4-bit): ~0.7 bytes × param_count → 8B model ≈ 8 GB (1× T4 16GB) Rule: Always leave 10-20% headroom for activation spikes and batch overhead.

Training Time Estimation

def estimate_training_time(num_examples, avg_tokens, epochs, batch_size, tps=5000): total_tokens = num_examples * avg_tokens * epochs total_hours = total_tokens / (batch_size * tps * 3600) return {"hours": round(total_hours, 1), "cost_a100": f"${total_hours * 3.50:.2f}", "cost_t4": f"${total_hours * 0.75:.2f}"} # Example: 1000 examples × 500 tokens × 3 epochs, batch=4 # → ~0.8 hrs, ~$2.80 on A100, ~$0.60 on T4
