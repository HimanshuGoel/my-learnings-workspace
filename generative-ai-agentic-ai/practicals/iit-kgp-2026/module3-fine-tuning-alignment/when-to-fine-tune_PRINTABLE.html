Module 3, Topic 1 — When to Fine-Tune

Fine-tuning modifies model weights by training on task-specific examples — it changes the model's DEFAULT behavior so it does things your way without being told. This is the most important topic in Module 3 because the #1 mistake practitioners make is fine-tuning when better prompting or RAG would have solved the problem at 1/100th the cost.

Problem: Before spending GPU hours and money, you need a rigorous framework to determine if fine-tuning is actually the right intervention — or if you're solving the wrong problem with an expensive hammer.

The Decision Ladder

Always climb this ladder in order. Jumping to Step 4 without exhausting Steps 1-3 is the #1 practitioner mistake.

What Fine-Tuning CAN vs CANNOT Fix

✓ CAN Fix (Behavior)

  • Format consistency (always output your JSON)
  • Tone/style (brand voice by default)
  • Domain terminology usage
  • Complex instruction following
  • Latency (shorter prompts needed)
  • Cost (no few-shot examples → fewer tokens)

✗ CANNOT Fix (Knowledge/Capability)

  • Knowledge gaps (use RAG instead)
  • Factual accuracy (doesn't make models truthful)
  • Reasoning ability (can't upgrade 7B → 70B)
  • Real-time information (knowledge still frozen)
  • Tasks the base model literally can't do
  • Safety issues (may make them worse)
Mental model: Fine-tuning changes HOW the model responds. RAG changes WHAT it knows. Prompting changes what you ASK for. Pick the intervention that matches the gap.

Failure Modes → Fixes

ProblemSymptomFix
OverfittingPerfect on training data, bad on new inputsMore diverse data, fewer epochs, add regularization
Catastrophic forgettingFine-tuned model loses general abilitiesUse LoRA (preserves base), mix in general data
Wrong diagnosisFine-tuning doesn't help the actual issueRevisit: is it behavior, knowledge, or capability?
Data quality issuesModel learns wrong patterns confidentlyAudit training data, remove contradictions/noise
No improvementMetrics flat after trainingDataset may not contain the signal (too small/uniform)
Cost overrunTraining takes longer than expectedStart with small experiments, estimate before scaling

Before proceeding to the rest of Module 3, confirm:
☐ I've tried 5+ prompt engineering iterations seriously
☐ I can articulate WHAT the model does wrong (format? style? domain? reasoning?)
☐ The gap is BEHAVIORAL (how it responds) not KNOWLEDGE (what it knows)
☐ I have 100+ examples of correct input→output
☐ The cost math works (training cost < 3 months of quality value)
☐ I've defined success criteria with measurable thresholds
☐ I accept this is an experiment (may need 2-3 iterations)
