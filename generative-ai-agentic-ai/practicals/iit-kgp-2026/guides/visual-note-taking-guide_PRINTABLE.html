Visual Note-Taking Guide

Practical reference for taking visual notes on paper while working with AI or studying new concepts.

Core Elements (Building Blocks)

ElementHow to DrawWhen to Use
Box / RectangleSimple bordered rectangleConcept, component, class, or system
Rounded boxRectangle with curved cornersProcess, action, or step
Circle / OvalEllipse shapeStarting/ending point, or central idea
DiamondRotated squareDecision or condition
Cylinder3D rectangle with curved topDatabase or storage
Cloud shapeWavy irregular blobExternal systems, APIs, internet
Stick figureCircle + linesUser or actor
Document iconRectangle with wavy bottomFile or document

Connectors (Relationships)

ConnectorHow to DrawMeaning
Solid arrow →Straight line with arrowhead"Flows to", "calls", "sends data to"
Dashed arrow -→Dashed line with arrowhead"Depends on", "references", "optional"
Double arrow ↔Arrows on both ends"Bidirectional", "communicates with"
Line (no arrow)Plain line"Is associated with", "belongs to"
Thick arrow ⇒Bold/thick arrow"Transforms into", "results in"
Dotted line ....Series of dots"Weak relationship", "might interact"

Techniques by Scenario

1. Understanding Architecture → Block Diagram

┌──────────┐ ┌──────────┐ ┌──────────┐ │ Client │──────→│ Server │──────→│ DB │ └──────────┘ └──────────┘ └──────────┘ │ ▼ ┌──────────┐ │ Cache │ └──────────┘
  • Draw each component as a box, arrows show data flow direction
  • Label arrows with what's sent (e.g., "HTTP request", "query")
  • Add boundary box (dashed rectangle) around grouped components

2. Learning a New Concept → Concept Map

┌─────────────┐ │ RAG │ └──────┬──────┘ ┌─────────┼─────────┐ ▼ ▼ ▼ ┌──────────┐ ┌────────┐ ┌──────────┐ │ Retrieve │ │Augment │ │ Generate │ └────┬─────┘ └────┬───┘ └────┬─────┘ ▼ ▼ ▼ Vector DB Prompt + LLM Output Context
  • Central concept at top/center, branch to sub-concepts
  • Add one-line definitions under each node
  • Use "is a", "has a", "uses" as arrow labels

3. Following a Process → Flowchart

┌───────┐ │ Start │ └───┬───┘ ▼ ┌───────────┐ │ Get Input │ └─────┬─────┘ ▼ ◇ Valid? ◇ / \ Yes No → ┌────────┐ ▼ │ Error │ ┌─────────┐ └────────┘ │ Process │ └────┬────┘ ▼ ┌──────┐ │ End │ └──────┘
  • Diamonds for decisions, keep happy path going down, branch sideways for errors
  • Number steps if sequence matters

4. Comparing Options → T-Chart / Comparison Grid

SQL vs NoSQL ┌──────────────┐ ┌──────────────┐ │ Structured │ │ Flexible │ │ ACID │ │ Eventual │ │ Joins │ │ Denormalized │ │ Scale ↕ │ │ Scale ↔ │ └──────────────┘ └──────────────┘ └────────── Use When ──────────┘ Fixed schema | Evolving schema Complex queries | Simple lookups

5. Understanding Code Flow → Sequence Diagram

User API Service DB │ │ │ │ │──request──→ │ │ │ │ │──validate──→ │ │ │ │ │──query──→ │ │ │ │←──data─── │ │ │←──result─── │ │ │←──response── │ │ │
  • Vertical lines = actors (name at top), horizontal arrows = messages
  • Read top to bottom for time sequence

6. Mental Models → Layered Diagram

┌─────────────────────────────────┐ │ Application Layer │ ├─────────────────────────────────┤ │ Framework Layer │ ├─────────────────────────────────┤ │ Runtime Layer │ ├─────────────────────────────────┤ │ OS / Hardware │ └─────────────────────────────────┘
  • Stack vertically: higher = more abstract, lower = more concrete
  • Add notes on side for "what lives here"

7. Brainstorming → Mind Map

┌── Vector DBs │ ┌── Storage ┤ │ └── Graph DBs │ AI ───┼── Models ─── Transformers │ └── Diffusion │ └── Tools ──── LangChain └── LlamaIndex
  • Central topic in middle, branch outward freely, no strict structure needed

Practical Tips

Layout Rules

  • Top-to-bottom for sequences and timelines
  • Left-to-right for data flows and pipelines
  • Center-outward for concept maps and mind maps
  • Leave whitespace — cramped notes are hard to read later
  • Consistent sizes — important things get bigger boxes

Making Them Look Good

  • Write labels in CAPS or underline headings
  • Keep text inside boxes short (2-4 words max)
  • Use indentation to show hierarchy
  • Draw borders/frames around complete diagrams

Pen Strategy (Multiple Colors)

  • Black: Structure, boxes, primary connections
  • Blue: Data flow, labels, annotations
  • Red: Errors, warnings, gotchas, decisions
  • Green: Happy path, success, key takeaways

Speed vs Neatness

  • During AI session: speed wins, messy is fine, capture relationships
  • After session (optional): spend 2 min redrawing the one best diagram — this redraw is itself a retention exercise

How Others Use Visual Notes

  • Software Engineers: Whiteboard architecture before coding, sequence diagrams on sticky notes, decision trees for library choices
  • Students (Cornell Method): Left col = key terms/boxes, Right col = visual relationships, Bottom = one-sentence summary
  • Sketchnote Practitioners: Combine text + icons + containers + connectors on single page, capture the "story"
  • Bullet Journal / Zettelkasten: One concept per card with small diagram, link cards with arrows/references, build network over time
