LlamaIndex — Complete Learning Notes

Summary: LlamaIndex is the database access layer for LLMs — it ingests, indexes, and queries your private data so LLMs can answer questions about it. Purpose-built for RAG.

Why LlamaIndex Exists

  • Problem — LLMs don't know your private data (docs, databases, internal wikis). Connecting them requires parsing, chunking, embedding, indexing, retrieving, and synthesizing — lots of plumbing.
  • Solution — a purpose-built data framework: load documents → create index → query with natural language. Five lines to a working RAG system.
  • Mental Model — ORM for LLMs (like Entity Framework or Hibernate). Just as an ORM abstracts SQL into objects, LlamaIndex abstracts chunking/embedding/retrieval into index.as_query_engine().query("question"). Or: if LangChain is Spring Framework (general), LlamaIndex is Spring Data (data-focused).
  • AI Connection — the fastest path to production RAG. Better retrieval quality out-of-the-box than manual LangChain RAG chains.

Where LlamaIndex Fits

Data → Readers → Nodes → Index → Query Engine → Answer (the LlamaIndex pipeline)

1. The 5-Line RAG

The fastest path from "I have documents" to "I can ask questions about them."

from llama_index.core import VectorStoreIndex, SimpleDirectoryReader docs = SimpleDirectoryReader("./my_docs").load_data() # 1. Load index = VectorStoreIndex.from_documents(docs) # 2. Index (embed + store) engine = index.as_query_engine() # 3. Create engine response = engine.query("What is the return policy?") # 4. Query print(response) # 5. Answer!
That's it. Behind those 5 lines: document parsing, chunking, embedding, vector storage, retrieval, prompt construction, LLM call, and response synthesis — all handled automatically.

Rule of thumb: Use LlamaIndex when your primary task is RAG (question answering over documents). Use LangChain when you need agents, tools, or complex multi-step orchestration. Use both together for production apps.

8. Common Mistakes

#MistakeFix
1Chunk size too large (imprecise retrieval)Try 256-512 tokens instead of default 1024
2Not persisting index (re-embeds every time)index.storage_context.persist()
3Using "compact" mode with many chunks (overflow)Switch to "refine" or "tree_summarize"
4Wrong similarity_top_k (too few=miss info, too many=noise)Start with 3-5, adjust based on results
5Not checking source_nodes (can't debug bad answers)Always inspect response.source_nodes
6One giant index for everythingSeparate indexes + router for different doc types

9. When NOT to Use

ScenarioUse InsteadWhy
General agent orchestrationLangChainLlamaIndex is data-focused, not agent-focused
Simple LLM call (no retrieval)Direct OpenAI SDKNo need for indexing infrastructure
Keyword search (exact match)ElasticsearchVector search ≠ keyword search
Model trainingHuggingFace / PyTorchLlamaIndex queries, doesn't train
Real-time streaming dataCustom pipelineLlamaIndex is for static/batch docs

Quick Reference

Load

SimpleDirectoryReader("./docs").load_data() Document(text="...", metadata={...})

Index

VectorStoreIndex.from_documents(docs) SummaryIndex.from_documents(docs)

Query

engine = index.as_query_engine( similarity_top_k=5, response_mode="compact") response = engine.query("question")

Chat

chat = index.as_chat_engine( chat_mode="condense_question") r = chat.chat("question") chat.reset()

Persist

index.storage_context.persist("./storage") index = load_index_from_storage( StorageContext.from_defaults( persist_dir="./storage"))

Response

response.response # answer text response.source_nodes # retrieved chunks node.text, node.metadata, node.score
