LangChain — Complete Learning Notes

Summary: LangChain orchestrates LLM-powered applications — chaining prompts, models, tools, memory, and retrieval into RAG pipelines and autonomous agents. It's Spring Framework for AI apps.

Why LangChain Exists

  • Problem — a raw LLM API call is stateless, has no access to your data, can't use tools, and produces unstructured text. Building AI apps requires orchestrating many components.
  • Solution — composable building blocks: chains (prompt→model→parser), retrievers (search your docs), agents (model decides actions), memory (conversation state).
  • Mental Model — Spring Framework for LLM Apps. Chains = middleware pipeline. Retrievers = data access layer. Agents = controllers with tool-calling. Memory = session state. Or: Express.js middleware — each step processes data and passes it along.
  • AI Connection — the standard framework for RAG systems, AI chatbots, and agentic applications.

Where LangChain Fits

LangChain orchestrates LLMs, retrieval, tools, and memory into complete applications

1. LCEL Chains — The Core Pattern

LCEL (LangChain Expression Language) uses the pipe operator to compose steps. This is the modern way to build everything.

LCEL: each step transforms data and passes to the next — like Express middleware
# Basic chain chain = prompt | llm | StrOutputParser() result = chain.invoke({"question": "What is AI?"}) # With retrieval (RAG) chain = {"context": retriever, "question": RunnablePassthrough()} | prompt | llm | parser

2. RAG Pipeline — The Core Architecture

RAG = Retrieval-Augmented Generation. Answer questions by first finding relevant documents, then generating with that context.

RAG pipeline: ingest documents once, then retrieve and generate for each query
# Complete RAG in ~10 lines docs = PyPDFLoader("doc.pdf").load() chunks = RecursiveCharacterTextSplitter(500, 50).split_documents(docs) vectorstore = Chroma.from_documents(chunks, OpenAIEmbeddings()) retriever = vectorstore.as_retriever(search_kwargs={"k": 3}) rag_chain = ( {"context": retriever | format_docs, "question": RunnablePassthrough()} | rag_prompt | llm | StrOutputParser() ) answer = rag_chain.invoke("What is the return policy?")

3. Agents — LLMs That Use Tools

  • What — an agent is an LLM that decides which tools to call and in what order, then synthesizes a final answer from tool outputs.
  • How — define tools with @tool decorator, create agent with create_tool_calling_agent(), execute with AgentExecutor.
  • When — when the model needs real-time info (search), calculations, database access, or multi-step reasoning.
@tool def search(query: str) -> str: """Search the web.""" return "search results..." agent = create_tool_calling_agent(llm, [search], prompt) executor = AgentExecutor(agent=agent, tools=[search], max_iterations=5) result = executor.invoke({"input": "Latest AI news?"})

6. Common Mistakes

#MistakeFix
1Using deprecated syntax (LLMChain, SequentialChain)Use LCEL pipe syntax for everything new
2Chunks too large or too smallStart with 500-1000 chars, 50-100 overlap
3Ignoring metadata in retrievalFilter by source, date, section for precision
4Agent infinite loopsSet max_iterations in AgentExecutor
5Not handling LLM rate limitsAdd .with_retry() on chains
6Over-engineering simple tasksSingle API call might be enough — don't force LangChain

7. When NOT to Use

ScenarioUse InsteadWhy
Simple single LLM callDirect OpenAI SDKLess overhead, simpler
Model training/fine-tuningHuggingFace TrainerLangChain doesn't train models
Latency-critical servingCustom code + vLLMLangChain adds abstraction overhead
Static FAQDirect vector DB queryNo need for chain orchestration

Top Takeaways (Ranked)

#Insight
1LCEL (prompt | llm | parser) is the one pattern to know — everything builds on it
2RAG = Load → Split → Embed → Store → Retrieve → Generate (memorize this flow)
3Agents let LLMs use tools — the model DECIDES what actions to take
4Chunk size matters: too big = imprecise retrieval, too small = lost context (start at 500)
5Always use LCEL pipe syntax — old chain classes are deprecated
6OutputParser + Pydantic = guaranteed structured JSON from LLMs
7Not everything needs LangChain — a direct API call might be simpler
8Retriever quality determines RAG quality — garbage in, garbage out
9Source attribution (returning which docs were used) is critical for trust
10LangChain wraps many providers (OpenAI, HuggingFace, Anthropic) — switch models without rewriting
