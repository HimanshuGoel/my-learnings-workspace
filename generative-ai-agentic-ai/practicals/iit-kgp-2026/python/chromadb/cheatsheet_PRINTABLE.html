ChromaDB — Complete Learning Notes

Summary: ChromaDB is SQLite for embeddings — a zero-config local vector database that stores, indexes, and retrieves text by semantic similarity. The storage layer for RAG systems.

Why ChromaDB Exists

  • Problem — after converting documents to embeddings, you need somewhere to store them and quickly find the most similar ones for a query. Recomputing on every query is expensive.
  • Solution — a vector database that stores embeddings + metadata + documents together, with fast approximate nearest-neighbor search. Zero config, local, just pip install.
  • Mental Model — SQLite for embeddings. As SQLite gives zero-config SQL storage, ChromaDB gives zero-config vector storage. SQL answers "WHERE condition matches." ChromaDB answers "WHERE meaning is similar."
  • AI Connection — the storage and retrieval layer in every RAG system. Without it, you'd search linearly through all embeddings (O(n)) on every query.

Where ChromaDB Fits

ChromaDB sits between embedding creation and LLM generation — the "database" in RAG

1. Core Operations

# Setup client = chromadb.PersistentClient(path="./db") # saves to disk collection = client.get_or_create_collection("docs") # Add documents (auto-embeds with default model) collection.add( ids=["id1", "id2"], documents=["text 1", "text 2"], metadatas=[{"source": "a"}, {"source": "b"}] ) # Query (find similar) results = collection.query(query_texts=["my question"], n_results=3) # → results["documents"], results["distances"], results["metadatas"] # Update collection.update(ids=["id1"], documents=["new text"]) # Delete collection.delete(ids=["id1"])

2. Metadata Filtering

Combine semantic similarity with structured filters — like adding WHERE clauses to your vector search.

OperatorMeaningExample
$eqEquals (default){"source": "manual.pdf"}
$neNot equals{"status": {"$ne": "draft"}}
$gt / $gteGreater (or equal){"page": {"$gte": 5}}
$lt / $lteLess (or equal){"score": {"$lt": 0.5}}
$in / $ninIn / not in list{"topic": {"$in": ["ml", "dl"]}}
$and / $orCombine filters{"$and": [{...}, {...}]}
# Filter + semantic search results = collection.query( query_texts=["deployment"], n_results=5, where={"source": {"$in": ["ops-guide.md", "deploy.md"]}} )

3. LangChain Integration

from langchain_community.vectorstores import Chroma from langchain_community.embeddings import HuggingFaceEmbeddings vectorstore = Chroma.from_documents(chunks, HuggingFaceEmbeddings(), persist_directory="./db", collection_name="docs") retriever = vectorstore.as_retriever(search_kwargs={"k": 3}) docs = retriever.invoke("my query") # → List[Document]

6. Common Mistakes

#MistakeFix
1Non-unique IDs (overwrite silently)Use content hashes or UUIDs as IDs
2Using in-memory client for real appsPersistentClient(path) for data survival
3Mismatched embedding dimensionsAlways use same model for store and query
4Not filtering by metadataUse where= to narrow before similarity search
5Storing long texts without chunkingSplit to 100-500 words first for precise retrieval
6Ignoring distance scoresSet threshold — low distance = high similarity

Memory Map

Setup ├── Client (in-memory or persistent) └── Collection (named group of embeddings) │ ▼ Ingestion ├── add() — insert new documents ├── upsert() — insert or update (prevents duplicates) └── Auto-embed or pre-computed embeddings │ ▼ Retrieval ├── query(query_texts) — semantic similarity search ├── where= — metadata filters ($eq, $gt, $in, $and) ├── where_document= — text content filter ($contains) └── n_results — how many to return │ ▼ Maintenance ├── update() — modify existing documents ├── delete() — remove by ID or filter └── get() — fetch by ID (non-semantic) │ ▼ Integration ├── LangChain: Chroma.from_documents() → retriever └── Raw: collection.query() → format → LLM prompt

Top Takeaways

#Insight
1ChromaDB is "SQLite for vectors" — zero config, local, pip install and go
2Core flow: add(docs) → query(question) → get similar docs (that's it)
3Metadata filtering is critical — always store source, page, topic for precise retrieval
4PersistentClient saves data to disk — use for anything beyond quick testing
5Distance = lower is better (0 = identical, higher = less similar)
6Always chunk documents before storing — 500 chars is a good default
7Same embedding model for storage AND query (mismatch = garbage results)
8Content hashes as IDs prevent duplicates on re-ingestion
