Pandas — Complete Learning Notes

Summary: Pandas is Python's tabular data toolkit — it loads, cleans, transforms, and explores structured data (CSV, Excel, SQL) using a spreadsheet-like API with method chaining.

Why Pandas Exists

  • Problem — real-world data comes in messy tables (CSV, Excel) with mixed types, missing values, and inconsistent formats. Processing it with raw Python requires hundreds of lines of manual parsing.
  • Solution — Pandas provides a DataFrame (labeled 2D table) with built-in methods for loading, filtering, grouping, joining, and transforming — like SQL + Excel + LINQ combined.
  • Mental Model — Pandas is Excel on steroids for developers. If NumPy is int[] (raw arrays), Pandas is DataTable in .NET — structured, labeled, mixed-type, queryable.
  • AI Connection — 80% of data science is data preparation. Every ML pipeline starts with Pandas for loading and cleaning before the model ever sees the data.

Where Pandas Fits in the AI Pipeline

Pandas is the universal data interchange format — everything starts here

Everything in Pandas is either a Series (1D column) or a DataFrame (2D table). Understanding this distinction is foundational.

A DataFrame is a dictionary of Series sharing the same index — like an Excel sheet in code

2. Loading & Inspecting Data

The first thing you do in any data project: load and understand what you have.

df = pd.read_csv("data.csv") # most common in ML df = pd.read_excel("data.xlsx") # business data df = pd.read_json("data.json") # API outputs # First look (run these 5 every time) df.shape # (rows, cols) — how big? df.head() # first 5 rows — what does it look like? df.info() # types + null counts — any issues? df.describe() # statistics — any outliers? df.isnull().sum() # missing values — how bad?
Rule of thumb: Always run these 5 commands first on any new dataset. Takes 30 seconds, saves hours of debugging.

3. Selection — loc vs iloc

The most important indexing distinction. Get this wrong and you'll have subtle bugs.

loc = label (inclusive), iloc = integer position (exclusive end) — the #1 source of Pandas confusion
# Boolean filtering — the most common pattern in ML df[df["age"] > 30] # simple filter df[(df["age"] > 25) & (df["salary"] > 60000)] # AND (use & not 'and') df[df["city"].isin(["NYC", "LA"])] # IN clause df.query("age > 25 and salary > 60000") # SQL-like string

4. Cleaning Data — The 80% of Data Science

Real data is messy. Missing values, duplicates, wrong types, inconsistent formats — Pandas handles all of it.

Cleaning pipeline: detect → fix → verify — run before every model
# Missing values df.isnull().sum() # count per column df.dropna(subset=["target"]) # drop if target is NaN df["age"].fillna(df["age"].median()) # fill with median (numeric) df["cat"].fillna("Unknown") # fill with string (categorical) # Duplicates df.drop_duplicates() df.drop_duplicates(subset=["email"]) # by specific column # Type fixes df["price"] = df["price"].astype(float) df["date"] = pd.to_datetime(df["date"]) df["name"] = df["name"].str.strip().str.lower() # clean strings
SettingWithCopyWarning trap: df[df["a"]>5]["b"] = 0 might not work! Use df.loc[df["a"]>5, "b"] = 0 instead. Always use .loc for assignment.

5. GroupBy — Split-Apply-Combine

The most powerful analysis pattern: split data into groups, apply a function to each, combine results. Equivalent to SQL GROUP BY.

GroupBy: split into groups → apply function to each → combine into result
# Basic groupby df.groupby("dept")["salary"].mean() # Multiple aggregations df.groupby("dept").agg( avg_salary=("salary", "mean"), headcount=("name", "count"), max_age=("age", "max") ) # Pivot table (Excel-style cross-tab) df.pivot_table(values="revenue", index="month", columns="product", aggfunc="sum")

6. Merging — SQL JOINs in Pandas

Combining data from multiple tables/sources. Exactly like SQL JOINs — same mental model applies.

Left join keeps all left rows — unmatched right side becomes NaN
# SQL-style joins pd.merge(left, right, on="key", how="inner") # only matching rows pd.merge(left, right, on="key", how="left") # keep all left rows pd.merge(left, right, on="key", how="outer") # keep all rows from both pd.merge(left, right, left_on="a", right_on="b") # different key names # Stack DataFrames pd.concat([df1, df2], axis=0) # rows (append) pd.concat([df1, df2], axis=1) # columns (side-by-side)

7. Feature Engineering for ML

Creating new features from existing data is where domain knowledge meets code. This directly improves model accuracy.

# Derived features (vectorized — no loops!) df["price_per_sqft"] = df["price"] / df["sqft"] df["age"] = 2026 - df["year_built"] df["is_premium"] = np.where(df["price"] > 500000, 1, 0) # Binning continuous → categorical df["age_group"] = pd.cut(df["age"], bins=[0,18,35,65,100], labels=["youth","young","mid","senior"]) # One-hot encoding (categorical → binary columns) df = pd.get_dummies(df, columns=["city", "type"], drop_first=True) # String extraction df["domain"] = df["email"].str.split("@").str[1] # Date features df["month"] = df["date"].dt.month df["is_weekend"] = df["date"].dt.dayofweek >= 5
ML rule: Raw categorical strings ("Red", "Blue") can't go into models. You must encode them first — pd.get_dummies() is the fastest way. Use drop_first=True to avoid multicollinearity (redundant columns that confuse the model).

8. Method Chaining — Fluent API Pattern

Chain operations like LINQ in C# or builder pattern in Java. Clean, readable, and avoids temporary variables.

# Instead of: temp = df[df["age"] > 25] temp = temp.groupby("dept").agg(avg=("salary","mean")) temp = temp.sort_values("avg", ascending=False) result = temp.head(5) # Use method chaining: result = ( df .query("age > 25") .groupby("dept") .agg(avg_salary=("salary", "mean")) .sort_values("avg_salary", ascending=False) .head(5) )

10. Common Mistakes

#MistakeFix
1SettingWithCopyWarning: chained indexing doesn't modify originalUse df.loc[mask, "col"] = value
2Forgetting inplace/reassignment: df.dropna() does nothingdf = df.dropna() or inplace=True
3Using apply() for simple math (slow Python loop)Use vectorized: df["x"] * 2
4Index gaps after filtering (0,1,5,8...)df.reset_index(drop=True)
5Loading huge file when you need 3 columnspd.read_csv(f, usecols=["a","b","c"])
6Using 'and'/'or' in boolean filter (Python logic vs element-wise)Use & / | with parentheses

11. When NOT to Use Pandas

ScenarioUse InsteadWhy
Pure number crunching (no labels needed)NumPyFaster, less overhead
Data larger than RAM (10GB+)Polars, Dask, PySparkPandas loads everything into memory
Maximum performance on transformsPolarsRust-based, 5-10x faster on large data
Real-time streamingKafka + FlinkPandas is batch-oriented
Graph/network dataNetworkXDataFrames aren't suited for graph traversal
Image/audio arraysNumPy directlyDataFrame overhead for pure arrays is wasteful

Mental Models Summary

  • DataFrame — Excel sheet in code / .NET DataTable
  • Series — one column (labeled dictionary with superpowers)
  • loc vs iloc — dict["key"] vs array[index]
  • Boolean filtering — SQL WHERE clause
  • GroupBy — SQL GROUP BY / LINQ GroupBy().Select()
  • Merge — SQL JOIN (inner, left, right, outer)
  • Concat — UNION ALL (axis=0) or side-by-side (axis=1)
  • Method chaining — builder pattern / fluent API
  • get_dummies — one-hot encoding for ML
  • apply() — Array.map() but slower (last resort)
  • fillna — imputation (fill gaps with best guess)
  • pivot_table — Excel pivot table in one line
