Module 5, Topic 3 — Containerization

Docker for AI services has unique challenges: large model files (GBs), GPU passthrough, multi-stage builds to keep images lean, and volume mounts for persistent vector stores. You know Docker — this focuses on AI-SPECIFIC patterns that prevent 10GB images, 30-minute builds, and OOM kills on startup.

Problem: ML Python packages (torch, transformers, sentence-transformers) are 2-5GB. Model files add 1-15GB. Naive Dockerfiles produce enormous, slow-building images. You need multi-stage builds, proper layer ordering, and a model-loading strategy.

Multi-Stage Dockerfile Flow

Multi-stage build: Stage 1 installs heavy ML deps (cached), Stage 2 copies only runtime packages + app code. No build tools in final image.

Deployment Topology (Docker Compose)

Docker Compose topology: API depends on ChromaDB + Redis. All services have health checks. Volumes persist data across restarts.

.dockerignore (Critical for AI)

Multi-Stage Dockerfile

# Dockerfile # Stage 1: Dependencies (cached — changes rarely) FROM python:3.11-slim AS dependencies WORKDIR /app RUN apt-get update && apt-get install -y --no-install-recommends build-essential \ && rm -rf /var/lib/apt/lists/* COPY requirements.txt . RUN pip install --no-cache-dir --upgrade pip && pip install --no-cache-dir -r requirements.txt # Stage 2: Runtime (lean — changes with your code) FROM python:3.11-slim AS runtime WORKDIR /app COPY --from=dependencies /usr/local/lib/python3.11/site-packages /usr/local/lib/python3.11/site-packages COPY --from=dependencies /usr/local/bin /usr/local/bin RUN apt-get update && apt-get install -y --no-install-recommends curl && rm -rf /var/lib/apt/lists/* COPY app/ ./app/ COPY prompts/ ./prompts/ RUN useradd -m -r appuser && chown -R appuser:appuser /app USER appuser HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \ CMD curl -f http://localhost:8000/health || exit 1 EXPOSE 8000 CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "2"]

GPU Docker Compose (Local Inference)

# docker-compose.gpu.yml — for serving local models (vLLM, HuggingFace) services: api: build: { context: ., dockerfile: Dockerfile.gpu } ports: ["8000:8000"] deploy: resources: reservations: devices: [{ driver: nvidia, count: 1, capabilities: [gpu] }] volumes: ["./models:/models"] # Mount models, don't bake into image! environment: - MODEL_PATH=/models/mistral-7b-instruct - CUDA_VISIBLE_DEVICES=0

✅ Containerization Checklist:
□ .dockerignore excludes: .git, .venv, models/, __pycache__, .env
□ Multi-stage Dockerfile: dependencies cached, app code in final stage
□ requirements.txt COPY before app code (layer cache optimization)
□ Run as non-root user (USER appuser)
□ HEALTHCHECK defined (60s start-period for model loading)
□ Models mounted as volume, NOT baked into image
□ docker-compose services have healthcheck + depends_on conditions
□ Volumes persist vector store and cache data across restarts
□ Image scanned for CVEs: docker scout cves ai-service
□ Specific image tags used (never :latest in production)
