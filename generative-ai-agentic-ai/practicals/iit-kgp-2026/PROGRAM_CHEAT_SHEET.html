Module 3 — Fine-Tuning: Key Decisions

DecisionDefault ChoiceWhen to Change
Fine-tune or not?DON'T (prompting + RAG first)Only if 5+ prompt iterations fail AND gap is behavioral AND you have 100+ examples
MethodLoRA (r=16, alpha=32)Full SFT only if LoRA demonstrably underperforms; QLoRA if GPU-limited
Dataset size200-500 examples (quality > quantity)50 for format only; 1000+ for complex reasoning
Learning rate2e-4 (LoRA), 2e-5 (full SFT), 5e-7 (DPO)Halve if unstable; double if loss doesn't decrease
Epochs3 (SFT/LoRA), 1 (DPO)Stop earlier if eval loss increases (overfitting)
Alignment methodDPO ($30-100, practical)RLHF only at scale ($5K-20K, production models for millions of users)
DPO β parameter0.1 (standard)Increase if model forgets capabilities; decrease if alignment too weak
Serving fine-tuned modelMerge LoRA → AWQ 4-bit → vLLMKeep LoRA separate for multi-task; bf16 for max quality
Fine-tuning golden rule: Prompting solves 95% of cases. Fine-tune changes HOW the model responds; RAG changes WHAT it knows. Don't confuse the two.

Cross-Module Decision Ladder

Problem: "Model doesn't do what I want" Step 1: Better prompting (free, hours) → Solves 70% Step 2: Add RAG (retrieve knowledge) → Solves 20% more Step 3: Prompt + RAG combined (max effort) → Solves 5% more Step 4: Fine-tuning (expensive, weeks) → Solves remaining 5% Step 5: Agentic (multi-step, tools) → For complex orchestration Step 6: Multi-agent (even more complex) → Almost never needed ALWAYS climb in order. Jumping to Step 4+ without exhausting 1-3 is the #1 mistake.
