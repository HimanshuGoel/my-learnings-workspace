Module 4, Topic 5 — Tool Use Design

Tools turn LLMs from text generators into action-takers. The LLM sees tool definitions (name, description, params), decides which to call, runtime validates args, executes, and returns results. Well-designed tool descriptions are THE key to reliable tool selection — the docstring IS your prompt to the LLM about when and how to use each tool. Poor descriptions lead to hallucinated parameters, wrong tool selection, and silent failures.

Problem Statement: An LLM without tools can only generate text. With tools, it can search databases, call APIs, execute code, and modify systems. But the interface between LLM and tools must be designed carefully — the LLM only sees the schema and description, so those must be precise enough for correct usage without human intervention.

Tool Execution Flow

Tool execution lifecycle: LLM decides → Runtime validates → Execute → Result back to LLM

Tool Contract Anatomy

Tool contract: name + docstring + Pydantic schema = everything the LLM uses to decide tool usage

Well-Designed Tool (with Pydantic Schema)

from langchain_core.tools import tool from pydantic import BaseModel, Field class SearchInput(BaseModel): query: str = Field(description="The search query") collection: str = Field(description="Which collection: 'docs', 'code', 'issues'") max_results: int = Field(default=5, ge=1, le=20, description="Number of results (1-20)") @tool(args_schema=SearchInput) def search_docs(query: str, collection: str, max_results: int = 5) -> str: """Search documentation, code examples, or GitHub issues. Use 'docs' for API documentation and guides. Use 'code' for code examples and implementations. Use 'issues' for bug reports and feature requests. Do NOT use for: general knowledge questions, current events.""" results = vector_store.similarity_search(query, k=max_results, filter={"collection": collection}) return "\n---\n".join([f"[{r.metadata['source']}] {r.page_content}" for r in results])

Error Wrapping (Critical Pattern)

@tool def query_database(sql: str) -> str: """Execute a read-only SQL query against the analytics database. Only SELECT queries are allowed. Returns results as formatted table.""" try: if not sql.strip().upper().startswith("SELECT"): return "ERROR: Only SELECT queries allowed. Rewrite as SELECT." result = db.execute(sql) if not result: return "Query returned no results. Broaden search criteria." return format_as_table(result) except SyntaxError as e: return f"SQL syntax error: {e}. Fix the query and try again." except TimeoutError: return "Query timed out (>30s). Add LIMIT or narrow WHERE clause." except Exception as e: return f"Database error: {type(e).__name__}: {e}. Try different approach." # KEY: Never raise — always return error STRING so agent can recover

Confirmation Pattern (Destructive Actions)

class DeleteAction(BaseModel): target: str = Field(description="What to delete") confirmed: bool = Field(default=False, description="Must be true to execute") @tool(args_schema=DeleteAction) def delete_resource(target: str, confirmed: bool = False) -> str: """Delete a resource. REQUIRES confirmed=true to actually execute. First call with confirmed=false to see preview.""" if not confirmed: return f"PREVIEW: Would delete '{target}'. Call again with confirmed=true." perform_delete(target) return f"Successfully deleted '{target}'."

Binding Tools to LLM + LangGraph

from langchain_openai import ChatOpenAI from langgraph.prebuilt import ToolNode llm = ChatOpenAI(model="gpt-4o-mini") tools = [search_docs, query_database, delete_resource] llm_with_tools = llm.bind_tools(tools) # In LangGraph: tool_node = ToolNode(tools) # Auto-executes tool calls from LLM graph.add_node("tools", tool_node) graph.add_conditional_edges("agent", should_continue, {"tools": "tools", END: END}) graph.add_edge("tools", "agent") # Result goes back to LLM

Architecture Decision: Design tools with explicit error messages (never raise exceptions to the LLM). When a tool raises unhandled exceptions, the agent can't recover. When it returns a helpful error STRING, the agent understands what went wrong and tries a different approach. This is the single most important tool design decision. Trade-off: more boilerplate per tool (try/except + formatted errors). Worth it for agent reliability.
