Module 4, Topic 7 — Planning & Reasoning

Planning lets agents decompose complex tasks into steps, execute them in order, and adapt when things go wrong. It adds 1-2 LLM calls overhead — only worth it for 3+ step tasks. Simple queries should bypass planning entirely via a complexity router. The core architecture is Plan → Execute → Replan, with a self-reflection variant for quality-sensitive outputs.

Problem Statement: Without planning, agents attempt everything in a single shot — works for simple queries but fails for multi-step problems requiring intermediate results, dependency ordering, or iterative refinement. Planning provides structure to multi-step reasoning but adds cost — the key is knowing when to plan vs when to skip.

Self-Reflection Loop

Self-reflection: Generate → Evaluate → Refine loop until quality threshold or max iterations

Implementation Code

Plan-and-Execute Agent

from langgraph.graph import StateGraph, END from langchain_openai import ChatOpenAI from pydantic import BaseModel, Field class Plan(BaseModel): steps: list[str] = Field(description="2-5 concrete steps") class PlanExecuteState(TypedDict): messages: Annotated[list, add] plan: list[str] current_step: int step_results: list[str] final_answer: Optional[str] iteration: int llm = ChatOpenAI(model="gpt-4o-mini") def planner_node(state: PlanExecuteState) -> dict: planner = llm.with_structured_output(Plan) response = planner.invoke([ SystemMessage(content="Break task into 2-5 concrete, actionable steps."), HumanMessage(content=f"Plan: {state['messages'][0].content}")]) return {"plan": response.steps, "current_step": 0, "step_results": []} def executor_node(state: PlanExecuteState) -> dict: step = state["plan"][state["current_step"]] previous = "\n".join(f"Step {i+1}: {r}" for i, r in enumerate(state["step_results"])) response = llm.invoke([ SystemMessage(content=f"Execute this step. Previous:\n{previous}"), HumanMessage(content=f"Execute: {step}")]) return {"step_results": [response.content], "current_step": state["current_step"] + 1, "iteration": state.get("iteration", 0) + 1} def should_continue(state: PlanExecuteState) -> str: if state.get("iteration", 0) >= 10: return "end" if state["current_step"] >= len(state["plan"]): return "synthesize" return "execute" graph = StateGraph(PlanExecuteState) graph.add_node("planner", planner_node) graph.add_node("executor", executor_node) graph.add_node("synthesize", synthesize_node) graph.set_entry_point("planner") graph.add_edge("planner", "executor") graph.add_conditional_edges("executor", should_continue, {"execute": "executor", "synthesize": "synthesize", "end": END}) graph.add_edge("synthesize", END)

Complexity Router (Skip Planning for Simple)

def complexity_router(state: dict) -> str: """Route simple queries directly, complex ones through planner.""" query = state["messages"][0].content response = llm.invoke([ SystemMessage(content="Reply SIMPLE or COMPLEX. SIMPLE = answerable in 1 step."), HumanMessage(content=query)]) return "direct" if "SIMPLE" in response.content else "planner" # In graph: entry → router → direct/planner graph.add_conditional_edges("router", complexity_router, {"direct": "direct_answer", "planner": "planner"})

Architecture Decision: Use Plan-and-Execute only for tasks with 3+ dependent steps. Route simple queries directly. Planning adds 1-2 LLM calls overhead — pure waste for "What's the weather?" but essential for "Research X, compare Y, write report." A cheap complexity router saves planning overhead on 70-80% of queries. Break-even: planning saves cost when it prevents wasted tool calls from wrong approaches.
