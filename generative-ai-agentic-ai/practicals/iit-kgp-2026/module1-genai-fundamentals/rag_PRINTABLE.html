Topic 10 — RAG (Retrieval-Augmented Generation)

Summary: RAG combines document retrieval with LLM generation — giving the model an open-book exam on YOUR data instead of relying on potentially outdated or hallucinated memory.

Why It Exists

  • Problem — LLMs have a knowledge cutoff (don't know recent events), can't access your private data, and hallucinate confidently when they don't know something.
  • Solution — before asking the LLM to answer, FIRST retrieve relevant documents from your data, then pass them as context. Answer grounded in evidence.
  • Mental Model — A developer answering questions using documentation. Instead of guessing from memory (hallucination risk), they search docs, read relevant sections, formulate answer, cite sources.
  • Or: Like dependency injection for knowledge. Instead of hard-coding facts into the model (fine-tuning), you inject relevant context at runtime (RAG).

How It Works

Two phases: Ingestion (index once) and Query (every question). Embed → Retrieve → Augment → Generate.

RAG architecture: ingest docs into vector DB (once), then embed→retrieve→augment→generate per query

RAG vs Fine-Tuning Decision Matrix

FactorRAGFine-Tuning
When data changesJust re-index (minutes)Retrain (hours/days)
Source attributionBuilt-in (show which docs)Not possible
Hallucination riskLower (grounded in docs)Higher (may still hallucinate)
LatencyHigher (retrieval step adds time)Lower (direct answer)
Private data handlingData stays in your DBData baked into model weights
Cost modelPer-query (retrieval + generation)One-time training + cheap inference
Rule of thumb: Use RAG when data changes frequently, attribution matters, or you can't afford fine-tuning. Use fine-tuning for consistent format/style with stable data. They're complementary — you can RAG with a fine-tuned model.

Key Design Decisions

DecisionOptionsTrade-off
Chunk size256 / 512 / 1024 tokensSmaller = precise retrieval, larger = more context per chunk
Chunk overlap0 / 50 / 100 tokensMore overlap = preserves boundary context, costs more storage
Top-K retrieval3 / 5 / 10 chunksMore = less risk missing info, but more noise and token cost
Embedding modelMiniLM (384d) / mpnet (768d) / OpenAI (1536d)Larger = better quality, slower + more expensive
Retrieval strategyVector only / Hybrid / Re-rankingMore stages = better precision, more complexity

The RAG Prompt Pattern

System: You are a helpful assistant. Answer ONLY based on the provided context. If the context doesn't contain the answer, say "I don't have that information." Context: {retrieved_chunk_1} {retrieved_chunk_2} {retrieved_chunk_3} Question: {user_question} Answer:
RAG = open-book exam. Model answers from retrieved evidence, not just memory. Reduces hallucination.

Common Misconceptions

WrongCorrect
"RAG = just copying text from documents"The LLM synthesizes, summarizes, and reasons over retrieved context.
"Bigger chunks = better"Too big = noise dilutes signal. 500 tokens is a good starting point.
"RAG eliminates hallucination"Reduces it significantly but doesn't eliminate. Model can still misinterpret context.
"RAG replaces fine-tuning"They solve different problems. RAG = knowledge access. Fine-tuning = behavior change.
"One retrieval step is enough"Production systems often re-rank, filter, and multi-step retrieve for quality.

Cosine similarityMeasure of how similar two vectors are (1 = identical, 0 = unrelated)
Re-rankingSecond-pass scoring of retrieved chunks for better precision
Hybrid searchCombining vector similarity with keyword matching for retrieval
