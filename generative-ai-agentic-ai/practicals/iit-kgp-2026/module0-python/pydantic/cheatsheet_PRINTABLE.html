Pydantic — Complete Learning Notes

Summary: Pydantic is TypeScript interfaces for Python — runtime data validation that ensures data from APIs, LLMs, and files matches expected structure and types. The type-safety layer for AI applications.

Why Pydantic Exists

  • Problem — Python is dynamically typed. Data from external sources (APIs, LLMs, files, user input) can be anything. Bugs from wrong types/shapes are caught too late.
  • Solution — declare data shapes as classes. Pydantic validates at creation time, coerces compatible types, and raises clear errors for invalid data.
  • Mental Model — TypeScript interfaces for Python. Like TS's interface User { name: string; age: number } but enforced at runtime. Also: Java DTOs with built-in validation.
  • AI Connection — forces LLMs to return structured JSON, validates API request/responses (FastAPI), defines tool schemas (function calling).

Where Pydantic Fits

Pydantic sits between untrusted external data and your application logic — the validation boundary

1. BaseModel — Define Data Shape

from pydantic import BaseModel, Field from typing import Optional, List class User(BaseModel): name: str # required age: int # required email: str is_active: bool = True # default value tags: List[str] = [] # default list bio: Optional[str] = None # nullable user = User(name="Alice", age=30, email="a@x.com") user = User(name="Bob", age="25", email="b@x.com") # "25" → 25 (coerced!)

2. Field Constraints

ConstraintTypeExample
gt, ge, lt, leNumbersField(gt=0, le=100)
min_length, max_lengthStringsField(min_length=1, max_length=50)
patternRegexField(pattern=r"^\d{3}$")
defaultAnyField(default="hello")
descriptionDocs/LLMField(description="user name")
For LLM schemas: Always include description in every Field. The LLM uses these descriptions to understand what each field means. Without them, it guesses (often wrong).

3. Serialization

# To dict / JSON user.model_dump() # → dict user.model_dump_json() # → JSON string user.model_dump(exclude={"email"}) # exclude fields # From dict / JSON User.model_validate({"name": "X", "age": 1, "email": "x@y.com"}) User.model_validate_json('{"name": "X", "age": 1, "email": "x@y.com"}') # JSON Schema (for APIs, LLMs, OpenAPI) User.model_json_schema() # → JSON Schema dict

4. Custom Validators

from pydantic import field_validator, model_validator class Order(BaseModel): quantity: int price: float @field_validator("quantity") @classmethod def check_qty(cls, v): if v < 1: raise ValueError("must be >= 1") return v @model_validator(mode="after") def check_total(self): if self.price * self.quantity > 10000: raise ValueError("order too large") return self

5. The AI Use Case — Structured LLM Output

The #1 reason Pydantic matters for AI engineers: forcing LLMs to return guaranteed JSON structures.

# Define what you want the LLM to return class RAGResponse(BaseModel): answer: str = Field(description="the answer to the question") confidence: float = Field(ge=0, le=1, description="confidence 0-1") sources: List[str] = Field(description="source document names") # LangChain integration from langchain_core.output_parsers import JsonOutputParser parser = JsonOutputParser(pydantic_object=RAGResponse) # LlamaIndex integration engine = index.as_query_engine(output_cls=RAGResponse) # OpenAI function calling schema = RAGResponse.model_json_schema() # pass as tool definition

6. FastAPI Integration

from fastapi import FastAPI class QueryRequest(BaseModel): question: str = Field(min_length=1) top_k: int = Field(default=3, ge=1, le=10) class QueryResponse(BaseModel): answer: str sources: List[str] @app.post("/query", response_model=QueryResponse) def query(request: QueryRequest): # auto-validated! return QueryResponse(answer="...", sources=["doc1"])

8. Common Mistakes

#MistakeFix
1Confusing v1/v2 syntax (dict() vs model_dump())Use v2: model_dump(), model_validate()
2No description in Field for LLM schemasAlways add description — LLM needs it
3Not handling ValidationErrortry/except when data is from external sources
4Over-constraining (LLM can't match strict regex)Relax constraints for LLM outputs
5Mutable default: default=[]Use default_factory=list
6Missing Optional for nullable fieldsOptional[str] = None if can be absent

Quick Reference

Define

class Model(BaseModel): field: type = Field(constraints) opt: Optional[type] = None lst: List[type] = [] lit: Literal["a", "b", "c"]

Create

Model(field=value) Model.model_validate(dict) Model.model_validate_json(str)

Output

inst.model_dump() # → dict inst.model_dump_json() # → JSON Model.model_json_schema() # → schema

Validate

@field_validator("name") @model_validator(mode="after") try: Model(...) except ValidationError as e: ...
