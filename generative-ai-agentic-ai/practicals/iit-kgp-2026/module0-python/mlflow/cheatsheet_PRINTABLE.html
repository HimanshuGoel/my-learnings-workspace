MLflow — Complete Learning Notes

Summary: MLflow is GitHub + Jenkins + SonarQube for ML — tracks experiments, versions models, and deploys them. The experiment lab notebook that's automated, searchable, and connected to production.

Why MLflow Exists

  • Problem — after training 50 models with different params, you can't remember which one worked, what data it used, or where the weights are. Results are lost in Jupyter output and scattered files.
  • Solution — automatic tracking of every experiment (params + metrics + model + artifacts), version control for models, and a UI to compare everything visually.
  • Mental Model — GitHub + Jenkins + SonarQube for ML. Tracking = commit history. Registry = releases (v1 staging, v2 production). Projects = Dockerfiles. Serving = CI/CD deploy.
  • AI Connection — how you prove your capstone works (reproducible experiments), compare fine-tuning runs, and manage model versions.

Where MLflow Fits

MLflow observes and records your training — it doesn't do the training itself

1. Tracking — Log Everything

import mlflow mlflow.set_experiment("my-experiment") with mlflow.start_run(run_name="rf_baseline"): mlflow.log_param("n_estimators", 100) mlflow.log_param("max_depth", 5) model.fit(X_train, y_train) mlflow.log_metric("f1", 0.87) mlflow.log_metric("accuracy", 0.92) mlflow.log_metric("loss", loss, step=epoch) # per-step mlflow.sklearn.log_model(model, "model") mlflow.log_artifact("confusion_matrix.png")

2. Auto-Logging (Zero Code)

mlflow.sklearn.autolog() # captures everything automatically model.fit(X_train, y_train) # params + metrics + model logged!
Always start with autolog. It captures params, metrics, and model for sklearn, PyTorch, and HuggingFace with zero manual calls. Add manual logging only for custom metrics.

Quick Reference

Tracking

mlflow.set_experiment("name") with mlflow.start_run(): mlflow.log_param("key", val) mlflow.log_metric("key", val, step=n) mlflow.log_artifact("file.png") mlflow.sklearn.log_model(model, "model")

Auto-log

mlflow.sklearn.autolog() mlflow.pytorch.autolog() mlflow.transformers.autolog()

Load Model

mlflow.sklearn.load_model("runs:/ID/model") mlflow.sklearn.load_model("models:/name/Production")

Search

runs = mlflow.search_runs( experiment_names=["exp"], filter_string="metrics.f1 > 0.85", order_by=["metrics.f1 DESC"])

UI

mlflow ui --port 5000 # http://localhost:5000

Memory Map

Tracking (Record) ├── set_experiment("name") ├── start_run(run_name="...") ├── log_param/log_params — hyperparameters ├── log_metric (+ step for curves) — results ├── log_artifact — files (plots, reports) └── log_model — model weights + metadata │ ▼ Comparison (Analyze) ├── mlflow ui — visual comparison dashboard ├── search_runs — programmatic filtering + sorting └── Find best model by metrics │ ▼ Registry (Manage) ├── Register model (give it a name) ├── Version: v1, v2, v3... ├── Stage: None → Staging → Production → Archived └── Load by name + stage for serving │ ▼ Deploy (Serve) ├── load_model("models:/name/Production") └── Use in FastAPI endpoint

Top Takeaways

#Insight
1MLflow tracks what you'd otherwise forget — which params, which data, which result
2autolog() captures everything with zero manual code for supported frameworks
3The UI (mlflow ui) lets you visually compare all experiments in one table
4Model Registry adds version control (v1, v2) and staging (Staging → Production)
5search_runs() returns a DataFrame — perfect for programmatic best-model selection
6Load models by name + stage: "models:/my-model/Production" (decoupled from run IDs)
7Log metrics per step for training curves (loss, accuracy over epochs)
8Your capstone evaluators will want reproducible experiment documentation — MLflow provides it
