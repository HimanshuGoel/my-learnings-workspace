Module 2, Topic 4 — Vector Store Design

A vector store is a specialized database for storing and querying embedding vectors via similarity search. Think of it as your relational database but instead of SQL JOINs, you do nearest-neighbor lookups in high-dimensional space. Module 1 covered what vector stores are. Here you're engineering the schema — collection design, metadata strategy, index tuning, and scaling plan.

Problem: Embeddings exist but "dump everything in one collection" breaks fast — no filtering, no multi-tenancy, slow at scale, no incremental updates. Vector store design = database schema design for RAG.

Store Selection Decision Tree

Pick based on scale and operational complexity. Start with ChromaDB for learning, graduate to Qdrant/pgvector for production.

Collection Architecture

Separate collections by document type. Each can have different chunk sizes, filters, and retrieval strategies.

Implementation — ChromaDB Setup

import chromadb client = chromadb.PersistentClient(path="./chroma_db") # survives restarts collection = client.get_or_create_collection( name="documentation", metadata={ "hnsw:space": "cosine", # distance metric (use cosine for normalized) "hnsw:M": 16, # connections/node (16 for <1M docs) "hnsw:construction_ef": 200, # build accuracy (higher = better index) "hnsw:search_ef": 100, # query accuracy (tune this for recall) } )

ID scheme: {doc_id}_chunk_{index} — enables incremental updates. Delete all chunks for a doc, insert new ones. Don't use random UUIDs (can't find chunks for a doc).

Querying with Filters

results = collection.query( query_embeddings=[query_embedding], n_results=5, where={"$and": [ {"doc_type": {"$eq": "api_doc"}}, {"version": {"$eq": "2.1.0"}}, ]}, where_document={"$contains": "authentication"}, # keyword filter )

Incremental Update Pattern

def upsert_document(collection, doc_id, chunks, metadatas, embeddings): # Delete old chunks for this document existing = collection.get(where={"source": doc_id}) if existing["ids"]: collection.delete(ids=existing["ids"]) # Insert new chunks ids = [f"{doc_id}_chunk_{i}" for i in range(len(chunks))] collection.add(ids=ids, documents=chunks, metadatas=metadatas, embeddings=embeddings)

HNSW Index Tuning

ParameterDefaultEffect of ↑Recommended
M (connections/node)16Better recall, more memory, slower build16 (<1M), 32 (>1M)
ef_construction200Better index quality, slower build200 (good default)
ef_search100Better query recall, slower queries100-200 (main tuning knob)
space/metricL2—cosine (for normalized embeddings)
ef_search is your query-time tuning knob. If recall@10 < 95%, increase it. Start at 100, go up to 300 before investigating other issues.
