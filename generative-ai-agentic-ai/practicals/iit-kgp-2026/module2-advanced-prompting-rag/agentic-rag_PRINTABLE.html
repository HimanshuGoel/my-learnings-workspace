Module 2, Topic 10 — Agentic RAG

Agentic RAG adds reasoning loops around retrieval — the system can plan which documents to fetch, verify that retrieved context is sufficient, retry with refined queries, and combine information across multiple retrieval steps. Think of it as upgrading from a single SQL query to a stored procedure that makes decisions. Module 1 introduced agents conceptually (ReAct, tool use). Here you're engineering the decision of WHEN to use agentic patterns vs simpler RAG, and HOW to implement them with LangGraph state machines.

Problem: Simple retrieve-then-generate fails on complex questions needing multi-step retrieval, cross-document reasoning, or dynamic routing. "Compare auth in v1 vs v2" needs multiple passes, decomposition, and synthesis.

When to Go Agentic — Decision Tree

80% of queries should use simple RAG. Only escalate when the question genuinely can't be answered in one pass.

Query Decomposition

class SubQueries(BaseModel): queries: list[str] = Field(min_length=2, max_length=4) strategy: str # How to combine sub-answers decomposer = llm.with_structured_output(SubQueries) result = decomposer.invoke(f"Break into 2-4 sub-questions: {question}") # "Compare auth v1 vs v2" → ["How does auth work in v1?", # "How does auth work in v2?", # "Migration steps from v1 to v2?"]

Self-RAG with LangGraph

from langgraph.graph import StateGraph, END graph = StateGraph(SelfRAGState) graph.add_node("retrieve", retrieve) graph.add_node("generate", generate) graph.add_node("check_grounding", check_grounding) # "Is answer in context?" graph.add_node("check_relevance", check_relevance) # "Does it answer Q?" graph.set_entry_point("retrieve") graph.add_edge("retrieve", "generate") graph.add_edge("generate", "check_grounding") graph.add_edge("check_grounding", "check_relevance") graph.add_conditional_edges("check_relevance", should_retry, { "accept": END, # Both checks pass → done "retry": "retrieve" # Failed → re-query (max 3 iterations) }) self_rag = graph.compile()

Agentic with Tools

from langgraph.prebuilt import ToolNode @tool def search_docs(query: str, collection: str = "default") -> str: """Search documentation.""" return "\n---\n".join([d.page_content for d in retriever.invoke(query)[:3]]) @tool def get_current_version(product: str) -> str: """Get current version of a product.""" return versions.get(product, "unknown") tools = [search_docs, get_current_version] llm_with_tools = ChatOpenAI(model="gpt-4o-mini").bind_tools(tools) # Agent decides: use tool or respond? Loop until done (max 10 steps)

Complexity Classifier (Route to Right Pattern)

def select_pipeline(question: str) -> str: if len(question.split()) < 10 and not any( w in question.lower() for w in ["compare", "vs", "difference", "steps"] ): return "simple_rag" # 80% of queries → fast path if any(w in question.lower() for w in ["compare", "vs", "difference"]): return "decomposed_rag" # Multi-part comparison if len(question.split()) > 20: return "decomposed_rag" # Complex question return "routed_rag" # Default middle ground

When NOT to Go Agentic

Don't use agentic RAG when:
• Simple factual lookup → single-shot is better (faster, cheaper, more reliable)
• High-throughput + low-latency requirements → agentic is too slow (5-30s)
• Can pre-compute answers (batch processing) → avoid real-time loops
• Corpus is small + well-organized → simple RAG with good chunking is enough
• You haven't maxed out simple RAG quality first → optimize retrieval/chunking before adding agents

Rule of thumb: If improving chunking or adding a reranker would fix the problem, don't reach for agentic.
