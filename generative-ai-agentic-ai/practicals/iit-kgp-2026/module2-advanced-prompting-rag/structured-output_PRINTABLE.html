Module 2, Topic 9 — Structured Output

LLMs are text generators, but downstream code needs structured data — JSON with specific fields, typed values, and validated constraints. Structured output enforcement makes LLMs behave like typed API endpoints instead of free-text generators. The key insight: define your schema as a Pydantic model (same as you'd use for FastAPI request/response validation), then use one of several enforcement mechanisms to guarantee the LLM conforms to it.

Problem: LLMs return free-form text but your app needs parseable JSON — typed fields, validated schemas. Without enforcement you get inconsistent formats, missing fields, invalid types, and brittle parsing.

Method Selection Decision Tree

Pick enforcement by provider + reliability needs. Pydantic model is your source of truth regardless of method.

Implementation — Step by Step

Step 1: Define Schema (Always Start Here)

from pydantic import BaseModel, Field from enum import Enum class Confidence(str, Enum): HIGH = "high" MEDIUM = "medium" LOW = "low" class Source(BaseModel): filename: str = Field(description="Source document filename") section: str = Field(description="Relevant section heading") relevance_score: float = Field(ge=0, le=1) class RAGAnswer(BaseModel): answer: str = Field(description="The answer to the user's question") confidence: Confidence = Field(description="Based on context quality") sources: list[Source] = Field(description="Sources used") follow_up_questions: list[str] = Field(default_factory=list, max_length=3) requires_human_review: bool = Field(default=False)

Schema Design Best Practices

DO ✓

  • Use Field(description="...") on every field
  • Use Enum/Literal for categorical values
  • Use Optional[] for uncertain fields
  • Keep nesting < 3 levels deep
  • Use list[...] with max_length
  • Add ge=/le= for numeric bounds
  • Version your schemas

DON'T ✗

  • Use Any or untyped dict
  • Nest > 3 levels (confuses models)
  • Use union types (ambiguous)
  • Make everything required (causes failures)
  • Use complex regex validators
  • Put instructions only in field name
  • Skip descriptions (model guesses wrong)
