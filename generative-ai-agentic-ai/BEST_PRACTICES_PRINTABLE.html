GenAI & Agentic AI — Best Practices

1. Prompt Engineering Fundamentals

High-Quality Prompt Structure

  • Role: Define the AI's persona (e.g., "expert frontend developer and accessibility specialist")
  • Task: Be specific about what is being built (e.g., "Create a responsive pricing card")
  • Constraints: Provide strict rules (e.g., "Use Tailwind CSS, use hex code #4F46E5, and no external dependencies")
  • Edge Cases: Explicitly ask for details like hover states or loading indicators

Key Principles

  • Don't use ChatGPT to "generate apps." Use it to think better. Explaining tradeoffs. Reviewing architectural ideas. Stress testing decisions. Drafting docs and explanations. It's like rubber-duck debugging, except the duck answers back.
  • Plus is worth it if you ask good questions. Useless if you don't. Write your problem clearly. Half the solution appears before the answer does.
  • A big task deserves a big prompt (see migration example below)
  • During prompt start including "Rewrite our title to attract the attention of our executive team"
  • Before you start, ask me any questions you need so I can give you more context. Be extremely comprehensive.
  • To avoid hallucination write prompt like "are there credible studies suggesting multitasking improves cognitive performance? if so, summarize them and note any limitations."

Big Task Prompt Example (Migration)

We need to migrate the frontend of this application from Blazor to React. The backend APIs and .Net Aspire orchestration must remain completely unchanged - only the rendering/UI layer changes. Before writing any code, I need you to: 1. Analyze the existing Blazor frontend architecture - identify all pages, components, routes and shared UI elements. 2. Map out every API dependency the frontend relies on (endpoints, auth flows, SignalR connections if any) 3. Produce a detailed migration plan including: proposed React project structure, state management approach, authentication flow and routing strategy. 4. Define a phased migration order that lets us validate incrementally - which slice do we migrate first and why? 5. Identify risks, unknowns and decisions that need input.

2. AI Coding Assistants

General

  • Claude Sonnet is the leader model to understand code.

Claude Code Skills (Reusable Instruction Files)

  • Keep it focused: One skill = one job. Avoid "do everything" skills. Smaller skills are more accurate and reusable.
  • Use clear role framing: Tell Claude exactly who it is acting as. Be specific (e.g., "UX researcher") instead of vague roles.
  • Think of it as a mini-program: Don't just write instructions. Define: inputs, outputs, steps. Treat it like structured logic, not a casual prompt.
  • Define clear input & output: Specify what data the skill expects, what result it should produce. Avoids confusion and randomness.
  • Use structured steps: Break the process into clear steps. Helps Claude follow a consistent workflow. Makes results predictable.
  • Keep it concise (avoid token bloat): Large skills = slower + worse performance. Only include what's necessary.
  • Test and refine: Try the skill on real tasks. Improve based on output quality. Iteration is key.

Copilot — Testing

Write a Playwright E2E test suite in tests/e2e/ that tests the current running Blazor app. Use accessible selectors and test user-visible behavior only - not Blazor-specific DOM.
